## このブログについて：爆速な技術ブログを作り直した理由

このブログは、エンジニアとしての学習・開発・失敗・改善の記録を残していくための個人技術ブログです。

名前は **「ともきちのエンジニア成長記」**。

実は、このブログ自体は今回が完全な新規作成ではありません。

もともと「ともきちのエンジニア成長記」は、Next.jsで作った技術ブログとして存在していました。

ただ、以前のブログはアニメーションを多用していたり、技術ブログとしては不要な機能をいろいろ入れていたりして、かなり重くなっていました。

見た目や動きにこだわること自体は悪いことではありません。

ただ、記事を読むための個人技術ブログとして考えた時に、本当にそこまで必要だったのかという疑問がありました。

さらに、React Server Componentsまわりの脆弱性対応が続いたことも、作り直しを考えるきっかけになりました。

もちろん、Next.js自体が悪いという話ではありません。

Webアプリでは、Next.jsの強力な機能はかなり役に立ちます。

ただ、更新頻度がそこまで高くなく、認証も課金もDBもAPIもほとんど必要ない個人技術ブログに、Next.jsを使い続ける必要が本当にあるのか。

そのことを考え直すタイミングが来たのだと思います。

そこで今回、以前のブログを少し直すのではなく、技術構成から見直して作り直すことにしました。

## なぜ作り直したのか

自分はこれまで、いくつかのブログやWebサービスを作ってきました。

旅行ブログの **「ともきちの旅行日記」** では、記事を書くこと、SEOを考えること、サイトを長く運用することの難しさを学びました。

AI旅程生成サービス **Tabidea** では、Next.js、Supabase、Cloud Run、Gemini API、Google Maps API、Stripe、CI/CD、RLS、認証、課金、パフォーマンスなど、より実践的なWebアプリ開発に触れてきました。

その中で、技術的な学びや設計の悩みはどんどん増えていきました。

ただ、それらは日々の会話やメモ、GitHubのissue、AIエージェントへの指示の中に散らばりがちです。

特に最近は、実装そのものだけでなく、

- なぜその技術を選んだのか
- なぜその設計にしたのか
- どこまでAIエージェントに任せるのか
- どこからは人間が判断すべきなのか
- 小さな個人開発でどこまで保守性や安全性を考えるべきなのか

といったことを考える機会が増えました。

そういう試行錯誤を、ちゃんと記事として残す場所が欲しくなりました。

だからこそ、ただ以前のブログを延命するのではなく、今の自分に合った形で作り直すことにしました。

## 以前はNext.jsで作っていた

以前の「ともきちのエンジニア成長記」はNext.jsで作っていました。

Next.jsはとても強力なフレームワークです。

ルーティング、画像最適化、API Routes、Server Components、動的レンダリング、認証、データ取得、キャッシュ戦略など、Webアプリを作るための機能が一通り揃っています。

TabideaのようなWebアプリでは、Next.jsはかなり相性が良いです。

一方で、このブログに必要なものはかなり少ないです。

基本的にはMarkdownで記事を書き、ビルド時にHTMLへ変換し、Cloudflareから高速に配信できれば十分です。

認証もいりません。

課金もいりません。

DBもいりません。

APIもほとんどいりません。

頻繁に更新するメディアでもありません。

そう考えると、Next.jsはこのブログには少し大きすぎると感じました。

Next.jsでも静的ブログは作れます。

ただ、使える機能が多いぶん、気づかないうちに設計がWebアプリ寄りになりやすいとも感じていました。

自分の以前のブログも、まさにそうでした。

アニメーションや機能を盛りすぎて、技術ブログとしては重くなっていました。

それなら、最初からコンテンツ配信に寄せた構成にした方がいい。

そう考えて、Next.js以外の選択肢を検討することにしました。

## Vite + Reactも候補だった

Next.jsを見直したあと、次に候補にしたのが **Vite + React** です。

Vite + Reactは以前にも使ったことがあり、開発体験も軽くて扱いやすい印象がありました。

Next.jsほど大きなフレームワークを使わず、自分で必要な部分だけ作るなら、Vite + Reactも良さそうだと考えました。

ただ、ブログとして必要なものを考えると、意外と自作する部分が多くなります。

例えば、

- Markdownの読み込み
- 記事一覧の生成
- タグページ
- 多言語対応
- RSS
- sitemap.xml
- OGPメタタグ
- 記事ごとのURL設計
- ビルド時のHTML生成

などです。

Reactで画面を作ること自体は簡単です。

でも、記事中心サイトとして必要な仕組みを整えようとすると、結局かなりの部分を自分で作ることになります。

さらに、Vite + Reactで作ると、基本的にはReactアプリとして考えがちです。

このブログでやりたいのは、インタラクティブなアプリ体験ではなく、速く、読みやすく、壊れにくい記事配信です。

そう考えると、Vite + Reactも今回の目的には少し違うかもしれないと思いました。

## 他の選択肢も考えた

Next.jsとVite + React以外にも、Reactまわりの選択肢はいくつか考えました。

その一つが **Remix** です。

RemixはWeb標準に寄せた設計や、ルーティング、データ取得、フォーム処理の考え方が魅力的なフレームワークです。

現在はReact Router v7系へ合流していく流れもあり、ReactでWebアプリを作る選択肢としてはかなり気になる存在でした。

ただ、今回作りたいのはWebアプリではなく、更新頻度も高くない個人技術ブログです。

フォームも認証も動的なデータ取得もほとんど必要ありません。

そのため、Remix系の強みを活かすよりも、もっとコンテンツ配信に寄った構成の方が合っていると判断しました。

**TanStack Router** や **TanStack Start** も、React文脈では気になる選択肢でした。

TanStack Routerは型安全なルーティングが魅力ですし、TanStack StartはViteとTanStack Routerを土台にしたフルスタックReactフレームワークとして注目しています。

ただ、今回のブログでは、そこまで高度なルーティングやフルスタック機能は必要ありません。

記事を書いて、ビルド時にHTMLへ変換し、Cloudflareから速く配信する。

必要なのはそれくらいです。

だから、TanStack系も今回の用途には少しアプリ寄りだと感じました。

Vue系であれば **Nuxt**、ほかにも **SvelteKit** のような選択肢もあります。

ただ、自分はこれまでReactとTypeScriptを中心に開発してきたため、このブログのためだけにVueやSvelteへ寄せる理由はあまりありませんでした。

また、Ruby on Rails、Django、Laravelのような他言語のWebフレームワークもありますが、自分はこれまでTypeScript以外のバックエンドフレームワークをほとんど触っていないため、今回の現実的な候補には入りませんでした。

結局、自分が今回必要としていたのは、フルスタックなWebアプリケーションフレームワークではありませんでした。

必要だったのは、Markdownの記事を扱いやすく、静的HTMLとして速く配信できて、必要な時だけ少し拡張できる仕組みです。

その条件に一番合っていたのがAstroでした。

## Astroを選んだ理由

最終的に選んだのが **Astro** です。

Astroは今回が初採用です。

最初から慣れていたわけではありません。

それでも、このブログにはAstroが一番合っていると感じました。

理由はシンプルです。

Astroは、コンテンツ中心のサイトと相性が良いからです。

Markdownを書いて、ビルド時にHTMLへ変換し、必要なページを静的に生成する。

このブログでやりたいことにかなり近いです。

特に大きかったのは、次の点です。

- Markdownとの相性が良い
- Astro Content Collectionsで記事を管理しやすい
- 静的HTMLを自然に生成できる
- 不要なJavaScriptを出しにくい
- 必要になった部分だけ後からインタラクティブにできる
- 多言語対応を設計しやすい
- Cloudflare Workers Static Assetsとの相性が良い
- AIエージェントに任せても構成が崩れにくい

このブログでは、記事本文をブラウザ側でMarkdownパースしません。

Markdownはビルド時にHTMLへ変換します。

ユーザーが記事を読む時には、すでに生成されたHTMLを受け取るだけです。

これにより、初期表示を軽くしやすく、JavaScriptに依存しない読みやすいページを作れます。

## このサイトの技術構成

このブログは、できるだけシンプルで速く、長く保守しやすい構成を目指しています。

主な技術構成は以下です。

- Astro
- Markdown
- Astro Content Collections
- TypeScript
- Tailwind CSS v4
- Biome
- pnpm
- Cloudflare Workers Static Assets
- GitHub Actions

ReactやMDXは使っていません。

このサイトでは、まずHTMLとCSSで読めることを優先しています。

JavaScriptは、必要な場所にだけ追加する方針です。

## ページ設計の考え方

このブログで大事にしているのは、記事をできるだけ素直なHTMLとして届けることです。

記事ページは、基本的に静的に生成されたHTMLとして配信します。

そのうえで、CSSで読みやすさを整え、必要な場合だけJavaScriptを足します。

特に意識しているのは、次のような方針です。

- 記事本文はビルド時にHTML化する
- クライアント側で記事データを組み立てない
- JavaScriptがなくても記事を読める状態にする
- JavaScriptは必要な機能にだけ使う
- 外部スクリプトやWebフォントを安易に増やさない
- Cloudflareからできるだけ速く配信する
- 機能追加よりも、表示速度と保守性を優先する

ページ遷移については、将来的にCSS View TransitionsやAstroのClientRouterなどを使って、体感速度を上げる余地はあります。

ただし、その場合でも本文配信の基本は静的HTMLのままにしたいです。

見た目のなめらかさを足すとしても、ページそのものを重くしすぎないことを重視します。

## これから入れる機能、入れない機能

このブログでは、機能をかなり絞っています。

今後追加する可能性があるのは、主に次の2つです。

- 検索機能
- シンタックスハイライト

記事数が増えれば、検索機能は必要になるかもしれません。

技術ブログなので、コードを書く記事が増えればシンタックスハイライトも必要になると思います。

一方で、それ以外の大きな機能は基本的に入れないつもりです。

例えば、

- コメント機能
- 会員登録
- CMS
- DB
- API
- 重いアニメーション
- 常設の複雑なインタラクション

などは、このブログにはあまり必要ないと考えています。

ただし、これは画像やアニメーション自体を否定しているわけではありません。

記事の理解に必要な図、スクリーンショット、比較画像などは使います。

必要な場所で軽いアニメーションを使う可能性もあります。

避けたいのは、記事を読むために必要ない常設機能や、雰囲気だけの重い演出を増やすことです。

以前のブログでは、そこを少し盛りすぎてしまいました。

今回はその反省を活かして、まずは読むことに集中できる構成にしています。

## 多言語対応について

このブログは、日本語をデフォルトにしつつ、英語版も用意します。

日本語の記事は通常のURLに置き、英語版は `/en/` 配下に置く構成です。

例えばこの記事なら、URLは次のようになります。

- `/articles/about-this-blog/`
- `/en/articles/about-this-blog/`

英語版を用意する理由は、単に海外向けに見せたいからだけではありません。

技術は英語圏の情報が多く、将来的に自分の考えや実装方針を英語でも説明できるようにしておくことは、エンジニアとして大事だと思っています。

また、AIエージェントと開発する機会が増えるほど、英語で設計意図を整理する力も重要になっていくはずです。

日本語で考えたことを英語でも説明できるようにする。

このブログは、その練習の場所にもしたいです。

## AIエージェントと開発する前提

このブログ自体も、AIエージェントに実装を任せやすい構成を意識しています。

最近の自分の開発では、Claude Code、Codex、Gemini CLIなどのコーディングエージェントを使うことが増えました。

ただ、AIに任せるほど、設計思想や制約を明文化しておくことが重要になります。

たとえばこのサイトでは、

- Reactを導入しない
- MDXを使わない
- 記事をクライアント側で描画しない
- APIやDBを持たない
- 不要な常設機能を増やさない
- まず静的HTMLとして速く読める状態を保つ

といった制約をあらかじめ決めています。

これは、AIが便利そうな機能を勝手に盛り込みすぎて、サイトの方向性がズレるのを防ぐためでもあります。

AIエージェントは実装を速く進めてくれます。

一方で、目的や制約が曖昧なままだと、必要以上に複雑な構成になることもあります。

だからこそ、人間側が「何を作らないか」「どこまでをサイトに持たせないか」を決めておくことが大事だと感じています。

## このブログで書いていくこと

このブログでは、主に次のようなことを書いていく予定です。

### 個人開発の設計と思考

Tabideaのような個人開発サービスを作る中で、どう設計を考えたのか、どこで迷ったのか、なぜその技術を選んだのかを書いていきます。

単なる実装手順だけでなく、判断の背景を残したいです。

### AIエージェントとの開発

Claude Code、Codex、Gemini CLIなどを使った開発の進め方、得意不得意、失敗例、指示書の作り方、レビューの仕方などを書きます。

AIにコードを書かせる時代だからこそ、人間側が何を考えるべきかを整理していきたいです。

### Webアプリのインフラと運用

Cloud Run、Cloudflare、Vercel、Netlify、Supabase、GitHub Actionsなど、個人開発で使うインフラやデプロイの話も書きます。

特に、無料枠や低コスト構成、学生・個人開発者にとって現実的な運用について書きたいです。

### フロントエンドとUX

Next.js、Astro、Tailwind CSS、パフォーマンス、エラー画面、待ち時間UX、多言語対応、ダークモードなど、ユーザー体験に関わる実装について書きます。

「AIを使ったからすごい」ではなく、実際に使いやすい体験をどう作るかを考えていきます。

### セキュリティと保守

Supabase RLS、認証、Cookie、OAuth、機密情報管理、CIでのチェックなど、個人開発でも避けて通れないセキュリティや保守の話も扱います。

小さなプロジェクトでも、最低限守るべきラインはあると思っています。

### 学習とキャリア

自分は、HTML/CSSの基礎から始めて、JavaScript、Next.js、個人ブログ、AIアプリ、クラウドデプロイへと少しずつ進んできました。

その過程で学んだことや、エンジニアとして働くために考えていることも書いていきます。

## 大事にしたいこと

このブログで大事にしたいのは、完成された正解を見せることではありません。

むしろ、

- 迷ったこと
- 失敗したこと
- 後から考えが変わったこと
- AIに任せてうまくいったこと
- AIに任せて危なかったこと
- 実装より設計が大事だと感じた瞬間

をちゃんと残していきたいです。

技術記事というと、きれいにまとまった結論だけを書きたくなります。

でも、実際の開発ではもっと泥臭い判断が多いです。

Next.jsで作っていたブログを、RSCの脆弱性対応をきっかけに見直す。

Vite + Reactも候補に入れる。

RemixやTanStack系など、React文脈の他の選択肢も考える。

そのうえで、今回の用途にはAstroが一番合っていると判断する。

こういう技術選定の過程も、自分にとっては大事な学びでした。

このブログでは、そういう過程も含めて記録していきます。
