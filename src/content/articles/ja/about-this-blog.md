# このブログについて：爆速な技術ブログを作った理由

このブログは、エンジニアとしての学習・開発・失敗・改善の記録を残していくために作った個人技術ブログです。

名前は **「ともきちのエンジニア成長記」**。

単に技術メモを置く場所というより、個人開発を続ける中で考えたこと、設計で迷ったこと、AIエージェントと一緒に開発して分かったこと、そして自分自身の成長過程を残していく場所にしたいと思っています。

## なぜ新しく作ったのか

これまでにもブログやWebサイトは作ってきました。

旅行ブログの **「ともきちの旅行日記」** では、記事を書くこと、SEOを考えること、サイトを長く運用することの難しさを学びました。

一方で、AI旅程生成サービス **Tabidea** では、Next.js、Supabase、Cloud Run、Gemini API、Google Maps API、Stripe、CI/CD、RLS、認証、課金、パフォーマンスなど、より実践的なWebアプリ開発に触れてきました。

ただ、そうした学びは日々の会話やメモ、GitHubのissue、AIエージェントへの指示の中に散らばりがちです。

そこで、技術的な考察や試行錯誤をきちんと記事として残すために、このブログを作りました。

## このサイトの技術構成

このブログは、できるだけシンプルで速く、長く保守しやすい構成を目指しています。

主な技術構成は以下です。

- Astro
- Markdown
- Astro Content Collections
- TypeScript
- Tailwind CSS v4
- Biome
- pnpm
- Cloudflare Workers Static Assets
- GitHub Actions

ReactやMDXは初期構成では使っていません。

このサイトはWebアプリではなく、**静的ドキュメント配信システム** として考えています。

記事を読むだけなら、重いクライアントJavaScriptは必要ありません。Markdownはビルド時にHTMLへ変換し、ブラウザ側でMarkdownをパースしないようにしています。

## Astroを選んだ理由

最初はViteとReactで自作の静的ブログを作る案も考えました。

ただ、今回作りたいものはアプリではなく、コンテンツ中心のブログです。

そのため、次の点を重視してAstroを採用しました。

- Markdownとの相性が良い
- 静的HTMLを自然に生成できる
- 不要なJavaScriptを出しにくい
- 多言語対応を設計しやすい
- 後から必要な部分だけインタラクティブにできる
- AIエージェントに実装を任せてもSPA化しにくい

特に大事だったのは、**「ブログをアプリにしすぎない」** ことです。

ページ遷移をSPAっぽくしたり、記事をクライアント側で描画したりすることもできますが、読むためのサイトとしては過剰になりやすいです。

だからこのブログでは、まず静的HTMLとして速く読めることを優先しています。

## デザインと機能の方針

このサイトでは、あえて初期機能をかなり絞っています。

初期実装では、以下のようなものは入れていません。

- 検索機能
- シンタックスハイライト
- 記事内画像
- コメント機能
- 会員登録
- API
- DB
- CMS
- Webフォント
- 重いアニメーション

一方で、最初から入れているものもあります。

- ダークモード
- 日本語・英語の多言語対応
- RSS
- sitemap.xml
- robots.txt
- llms.txt
- OGPメタタグ
- 共通OGP画像

検索やシンタックスハイライトは、記事数が増えてから必要になれば追加すればいいと考えています。

最初から全部入れるより、まずは軽く、速く、壊れにくい状態で始める方が自分には合っていると思いました。

## 多言語対応について

このブログは、日本語をデフォルトにしつつ、英語版も用意します。

日本語の記事は通常のURLに置き、英語版は `/en/` 配下に置く構成です。

例えばこの記事なら、URLは次のようになります。

- `/articles/about-this-blog/`
- `/en/articles/about-this-blog/`

英語版を用意する理由は、単に海外向けに見せたいからだけではありません。

技術は英語圏の情報が多く、将来的に自分の考えや実装方針を英語でも説明できるようにしておくことは、エンジニアとして大事だと思っています。

また、AIエージェントと開発する機会が増えるほど、英語で設計意図を整理する力も重要になっていくはずです。

## AIエージェントと開発する前提

このブログ自体も、AIエージェントに実装を任せやすい構成を意識しています。

最近の自分の開発では、Claude Code、Codex、Gemini CLIなどのコーディングエージェントを使うことが増えました。

ただ、AIに任せるほど、設計思想や制約を明文化しておくことが重要になります。

たとえばこのサイトでは、

- Reactを初期導入しない
- MDXを使わない
- 記事をクライアント側で描画しない
- 画像や検索を不用意に増やさない
- まず静的HTMLとして速く読める状態を保つ

といった制約をあらかじめ決めています。

これは単なる縛りではなく、AIが勝手に便利機能を盛りすぎてサイトの方向性がズレるのを防ぐための設計です。

## このブログで書いていくこと

このブログでは、主に次のようなことを書いていく予定です。

## 個人開発の設計と思考

Tabideaのような個人開発サービスを作る中で、どう設計を考えたのか、どこで迷ったのか、なぜその技術を選んだのかを書いていきます。

単なる実装手順だけでなく、判断の背景を残したいです。

## AIエージェントとの開発

Claude Code、Codex、Gemini CLIなどを使った開発の進め方、得意不得意、失敗例、指示書の作り方、レビューの仕方などを書きます。

AIにコードを書かせる時代だからこそ、人間側が何を考えるべきかを整理していきたいです。

## Webアプリのインフラと運用

Cloud Run、Cloudflare、Vercel、Netlify、Supabase、GitHub Actionsなど、個人開発で使うインフラやデプロイの話も書きます。

特に、無料枠や低コスト構成、学生・個人開発者にとって現実的な運用について書きたいです。

## フロントエンドとUX

Next.js、Astro、Tailwind CSS、パフォーマンス、エラー画面、待ち時間UX、多言語対応、ダークモードなど、ユーザー体験に関わる実装について書きます。

「AIを使ったからすごい」ではなく、実際に使いやすい体験をどう作るかを考えていきます。

## セキュリティと保守

Supabase RLS、認証、Cookie、OAuth、機密情報管理、CIでのチェックなど、個人開発でも避けて通れないセキュリティや保守の話も扱います。

小さなプロジェクトでも、最低限守るべきラインはあると思っています。

## 学習とキャリア

自分は、HTML/CSSの基礎から始めて、JavaScript、Next.js、個人ブログ、AIアプリ、クラウドデプロイへと少しずつ進んできました。

その過程で学んだことや、エンジニアとして働くために考えていることも書いていきます。

## 大事にしたいこと

このブログで大事にしたいのは、完成された正解を見せることではありません。

むしろ、

- 迷ったこと
- 失敗したこと
- 後から考えが変わったこと
- AIに任せてうまくいったこと
- AIに任せて危なかったこと
- 実装より設計が大事だと感じた瞬間

をちゃんと残していきたいです。

技術記事というと、きれいにまとまった結論だけを書きたくなります。

でも、実際の開発ではもっと泥臭い判断が多いです。

このブログでは、そういう過程も含めて記録していきます。

## おわりに

このブログは、まだ始まったばかりです。

最初から大きなメディアにするつもりはありません。

まずは、自分が学んだこと、考えたこと、作ったもの、失敗したことを、静かに積み上げていく場所にしたいと思っています。

そして後から振り返った時に、エンジニアとしての成長の跡が見えるようなブログになればいいなと思っています。
